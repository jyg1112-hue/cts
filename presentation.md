**발표자:** 조용근  
**발표 시간:** 7분 (질의응답 3분)

# 1. 문제 정의

---

## 우리 팀의 업무 방식

### 하역 데이터를 어떻게 쓰고 있었나?

- 선석별 하역 실적은 **엑셀 파일**로 관리
- 선박별 특이사항·이슈·정비 내역은 담당자 개인이 정리·보유
- 데이터는 있지만 → **꺼내 보는 게 번거롭다**

> "이슈 있는 선박의 평균 하역률이 어떻게 됐지?"
> 
> → 엑셀 열고 → 필터 걸고 → 계산하고 → 5~10분 소요

---

## 문제가 중요한 이유

### 수치로 본 비효율

| 구분 | 기존 방식 | AI 챗봇 |
|------|-----------|---------|
| 수치 조회 (예: 3월 평균 하역률) | 엑셀 필터 + 수동 계산, **5~10분** | **5초** |
| 이슈 원인 파악 (예: CSU 정비 경위) | 담당자 문의 or 파일 탐색 | **5초** |
| 복합 질의 (이슈 조건 + 수치) | 불가능에 가까움 | **가능** |

- **60~120배** 조회 시간 단축
- 담당자 부재 시에도 **누구나** 데이터 접근 가능

---

## 문제를 바라본 관점

### 가설: "데이터가 없는 게 아니라 꺼내기 어려운 것이다"

- 실적 데이터(수치)와 운영 노하우(정성적 이슈)가 **분리**되어 있음
- 두 데이터를 **함께 조회·연결**할 수 있으면 의사결정 속도가 빨라진다
- 자연어로 물어보는 방식이 가장 낮은 진입장벽

→ **"말로 물어보면 바로 답하는 하역 데이터 챗봇"** 을 핵심으로 설정

## 시스템 전체 구성

```
[사용자]
   │
   ▼
[인증 레이어] ← platform_auth.py (세션 쿠키)
   │
   ├─ [하역 챗봇] ─────────────────────────────────────┐
   │     └─ LangGraph 라우터                            │
   │          ├─ SQL 체인  (수치 집계)                  │
   │          ├─ RAG 체인  (이슈·원인 탐색)             │
   │          └─ Hybrid    (이슈 조건 + 수치)           │
   │               └─ BM25 + ChromaDB + Reranker        │
   │                                                    │
   ├─ [공급망 뉴스]                                     │
   │     └─ Tavily API → Google RSS fallback            │
   │          └─ OpenAI 한국어 요약                     │
   │                                                    │
   ├─ [야드 시뮬레이션] ← yard.html + Chart.js          │
   └─ [선석 스케줄]    ← schedule.html                  │
                                                        │
[FastAPI 백엔드] ←──────────────────────────────────────┘
[React 프론트엔드 (확장 중)]
```

## 핵심 기능 ① — 하역 챗봇 (AI 질의 엔진)

### 3단계 라우팅 구조 (LangGraph)

| 질문 유형 | 처리 경로 | 예시 |
|-----------|-----------|------|
| 수치·순위·목록 | **SQL 체인** | "3월 평균 하역률은?" |
| 이슈·원인·정비 | **RAG 체인** | "CSU2호 정비 경위는?" |
| 이슈 조건 + 수치 | **Hybrid** | "돌발정비 선박의 평균 하역률은?" |

**하이브리드 검색 (RAG 내부)**
- BM25 (키워드) + ChromaDB (벡터 임베딩) → RRF 병합
- Cross-Encoder Reranker로 최종 5개 문서 선별
- 메타데이터 필터: 월/연도/선박명/화물종류/이슈 유형

---

## 핵심 기능 ② — 그 외 주요 기능

### 공급망 뉴스
- 니켈·석탄 관련 최신 뉴스 자동 수집
- Tavily 우선 → Google RSS fallback
- OpenAI로 한국어 제목·요약 생성 (캐시 TTL 15분)

### 야드 시뮬레이션
- 야드 구역별 재고 현황 시각화 (Chart.js)
- 총재고 추이 차트 토글 기능

### 인증 및 접근 제어
- 세션 쿠키 기반 로그인/로그아웃
- 계정 관리 (추가·삭제·비밀번호 변경)
- 감사 로그 기록 (IP·사용자·시각)

## 교육 내용과의 연계

### 적용한 개념들

- **RAG (Retrieval-Augmented Generation)**: 문서 검색 + LLM 답변 생성
- **벡터 임베딩 & 유사도 검색**: ChromaDB를 통한 의미 기반 검색
- **LangGraph**: 상태 기반 에이전트 흐름 설계
- **프롬프트 엔지니어링**: 라우팅 분류, 응답 포맷 제어
- **API 설계**: RESTful 엔드포인트, 미들웨어

### 선택 이유 및 대안 검토
- 단순 LLM 호출만으로는 **수치 집계 정확도** 한계 → SQL 체인 추가
- 벡터 검색만으로는 **키워드 매칭 누락** → BM25 하이브리드 채택
- 단일 체인으로는 복합 질의 처리 불가 → LangGraph 상태 그래프 도입

---

## 시행착오 과정

### 버전 1: 단순 엑셀 업로드 + 챗봇

> 엑셀 데이터를 통째로 LLM에 전달 → 단순 질문은 답변 가능

**문제:** "수분 이슈가 있었던 선박의 평균 하역률은?" 같은 복합 질문에서 **완전히 실패**

---

### 버전 2: SQL만 추가

> 수치 집계는 해결됨

**문제:** 비고란의 정성적 이슈 데이터 (정비 경위, 화물 상태 등) 를 SQL로 표현 불가

---

### 버전 3: SQL + RAG 분리 라우팅 → **현재 구성**

- Claude와 반복 피드백: 실제 질문 예시를 넣어보며 어느 노드로 이어지는지 확인
- 라우팅 분류 프롬프트를 여러 번 수정해 경계 케이스 처리
- Hybrid 노드를 별도로 분리해 "이슈 조건 + 수치" 쿼리 처리

**핵심 학습:** 데이터의 성격(정량 vs 정성)에 따라 처리 경로를 나눠야 한다

---

## 멘토링 피드백 반영

### 받은 피드백과 개선 내용

| 피드백 주제 | 내용 | 반영 결과 |
|-------------|------|-----------|
| **데이터 저장 구조** | 챗봇 대화 데이터를 어떻게 저장할지 | SQLite 기반 구조 설계 |
| **쿼리 구성 방법** | 어떤 방식으로 챗봇 질의를 설계하면 좋을지 | 라우팅 프롬프트 + SQL 체인 구성 |
| **배포 기술** | 정규 교육에 없었던 Docker, Vercel 배포 개념 | Dockerfile + vercel.json 구성 완료 |

> 교육 커리큘럼 외의 영역(배포, 인증)도 질문하며 실제 서비스 수준으로 완성도를 높임

## 기대 효과

### 정량적 효과

| 항목 | 개선 수치 |
|------|-----------|
| 데이터 조회 시간 | 5~10분 → **5초** (약 60~120배 단축) |
| 복합 질의 처리 가능 여부 | 불가 → **가능** |
| 데이터 접근 가능 인원 | 담당자 1인 → **전체 팀원** |

### 정성적 효과

- **지식 공유**: 담당자만 알던 운영 노하우가 시스템화
- **의사결정 속도**: 회의 중 실시간 데이터 조회 가능
- **업무 집중도**: 반복 조회 업무에서 해방 → 핵심 업무 집중
- **확장성**: 다른 선석·품종 데이터도 동일 구조로 적용 가능

## 인사이트 및 회고

### 얻은 통찰

- **"데이터 구조가 AI 품질을 결정한다"**  
  정성 데이터를 얼마나 잘 정리해뒀느냐가 RAG 답변 품질의 핵심
  
- **"도구가 아니라 문제부터"**  
  어떤 기술을 쓸지보다 어떤 질문에 답해야 하는지를 먼저 정의해야 한다

- **"AI는 대화하면서 만든다"**  
  Claude와 반복 피드백하며 설계를 다듬는 과정 자체가 개발 방법론이 됨

### 아쉬운 점과 배운 가치

- 초기 단순 구조가 실패 → **덕분에 라우팅 설계의 필요성을 몸으로 이해**
- 배포·인증은 교육 범위 밖 → **스스로 찾고 적용하며 실무 감각 획득**
- React 프론트엔드는 아직 진행 중 → 완성보다 **동작하는 핵심 기능 우선** 원칙 학습

## 교육 소감 및 향후 계획

### 한 달간 가장 크게 성장한 것

> **"바이브 코딩을 알게 된 것"**

- AI 기본 개념(RAG, 임베딩, 프롬프트)에 익숙해진 것도 있지만
- AI와 **대화하며 만드는 개발 방식 자체**를 체득한 것이 가장 큰 수확

### 실무 활용 계획

앞으로 이런 상황에 바로 적용할 수 있다:

| 상황 | 활용 방식 |
|------|-----------|
| 반복적인 단순 업무 | 자동화 스크립트 + AI 처리 |
| 병목 구간 발생 | 데이터 기반 원인 분석 챗봇 |
| 비정형 데이터 정리 | LLM 기반 구조화 파이프라인 |
| 복합 이슈 분석 | Hybrid RAG로 정황·수치 연결 |
| 데이터 기반 의사결정 | 자연어 질의 → 실시간 인사이트 |

> **AI 활용은 특별한 프로젝트가 아니라 일상 업무 도구가 될 것이다**
